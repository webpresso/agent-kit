# Hook Matrix

This matrix separates host-documented lifecycle capability from what webpresso
currently scaffolds and verifies in this repo.

## Routing ownership

- Agent-kit owns `wp_*` dev-workflow routing and `wp_session_*` context-window-protection routing.
- `.omx/` is runtime/state only. It is not a direct hook surface.
- Release-facing replacement parity claims are gated by
  [`docs/bench/reference-parity-matrix.md`](bench/reference-parity-matrix.md)
  and crosswalked to the canonical host lifecycle support levels in
  `src/cli/commands/init/scaffolders/agent-hooks/capability-matrix.ts`.
  A replacement row cannot claim `full` host parity when any covered host/event
  is `partial`, `unmapped`, or `unsupported`.

## Claude Code

**Host-documented surface**

- Claude hooks live in `.claude/settings.json`.
- Supported events used in this hooks-orchestrator series include
  `SessionStart`, `PreToolUse`, `PostToolUse`, `PostToolUseFailure`,
  `UserPromptSubmit`, `Stop`, `PermissionRequest`, `SubagentStart`,
  `SubagentStop`, `SessionEnd`, `PreCompact`, and `PostCompact`.
- `Edit`, `Write`, and `MultiEdit` are distinct Claude tool matchers.

**Agent-kit scaffolds and verifies**

- `wp-sessionstart-routing`
- `wp-pretool-guard`
- `wp-post-tool`
- `wp-guard-switch`
- `wp-precompact-snapshot`
- `wp-stop-qa`
- `worktree.symlinkDirectories` includes `.claude`
- Claude `PreToolUse` and `PostToolUse` matchers cover `Bash`, `Read`, `Grep`, `WebFetch`, `Agent`, edit tools, `MultiEdit`, and MCP tool names so session-memory routing/capture sees context-heavy operations.

## Codex

**Host-documented surface**

- Codex supports project hook configuration via `.codex/hooks.json`.
- The documented lifecycle webpresso tracks here includes `SessionStart`,
  `PreToolUse`, `PostToolUse`, `Stop`, `PermissionRequest`, `SubagentStart`,
  `SubagentStop`, `PreCompact`, and `PostCompact`.
- We do not currently rely on richer per-tool matcher claims beyond the hook file
  contract itself.

**Agent-kit scaffolds and verifies**

- `.codex/hooks.json` is generated by `wp setup`; in the agent-kit source repo, `WP_FORCE_SOURCE=1 wp setup repair` emits direct commands with `WP_FORCE_SOURCE=1` so hook dispatch is JIT/source-first
- events: `SessionStart`, `PreToolUse`, `PostToolUse`, `UserPromptSubmit`, `Stop`, `PreCompact`
- PreToolUse matcher contract includes `Bash|apply_patch|Edit|Write|mcp__.*`; PostToolUse capture is broad for Codex-supported hooks.
- richer native events are tracked in the capability matrix as
  `partial`/`unmapped` until this repo actually emits and verifies them, and
  they remain accepted by `wp hooks dispatch` / `wp hooks demo` even when the
  current managed `wp-*` surface does not install a dedicated hook for each one

**Trust sync**

After writing `.codex/hooks.json`, webpresso asks the installed Codex runtime for
hook identity instead of computing hook hashes itself. It starts
`codex app-server --listen stdio://`, calls `hooks/list` for the repo, filters the
result to webpresso-owned unmanaged command hooks, then writes those hooks'
`key` + `currentHash` values to `hooks.state` through `config/batchWrite`.

Only webpresso-owned hooks are auto-trusted. A discovered hook must come from the
expected webpresso Codex hook source, be an unmanaged command hook, have no plugin
owner, and invoke one of the known `wp-*` hook bins. User-authored hooks, managed
hooks, plugin hooks, and unrelated commands remain for normal Codex review.

If `codex` is missing, `codex app-server` fails, the protocol call times out, or
post-write verification still reports untrusted/disabled hooks, setup prints one
concise warning and leaves the hooks reviewable in Codex with `/hooks`.

**Explicit non-claim**

- webpresso does **not** currently claim Codex `MultiEdit` parity
- webpresso does **not** currently claim richer post-edit automation beyond the
  generated `Edit|Write` hook surface
- webpresso does **not** treat third-party protocol crates as release proof
- webpresso does **not** claim full replacement parity for Cursor/OpenCode
  lifecycle coverage while the capability matrix marks covered events as
  partial, unmapped, or unsupported

## Typed continuity and release proof

The managed continuity lane uses `SessionStart` for restore/injection and typed
continuity events for post-tool capture, prompt-decision capture, stop summaries,
and PreCompact snapshots where the host emits that lifecycle. The hard-cut store schema rejects
untyped event rows; docs and package gates must not describe inferred event
semantics.

Current release-facing host boundary:

| Host     | Continuity depth                                                                                            | Release note                                                                                           |
| -------- | ----------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------ |
| Claude   | full for replacement-critical managed lifecycle events                                                      | Active hooks live in `.claude/settings.json`; `.claude-plugin/` is packaged metadata.                  |
| Codex    | full for emitted managed lifecycle events; richer native-only events remain partial/degraded in this matrix | Active hooks live in `.codex/hooks.json`; `.codex-plugin/` is packaged metadata.                       |
| Cursor   | degraded                                                                                                    | No managed PreCompact projection; capability rows remain partial or unsupported where pinned by tests. |
| OpenCode | degraded                                                                                                    | Plugin bridge covers selected lifecycle callbacks and does not emit the full managed hook set.         |

Run these release gates before hook-bin, continuity-doc, or package-surface
claims move forward:

```bash
./bin/wp hooks doctor --skip-mcp
./bin/wp audit blueprint-lifecycle
./bin/wp audit reference-parity-matrix --json
./bin/wp audit package-surface
npm pack --dry-run --json
vp run lint:pkg
vp run verify:secrets
./bin/wp audit secrets-policy
./bin/wp audit no-dev-vars
./bin/wp audit secret-provider-quarantine
./bin/wp audit secrets-config
vp run verify:paths
```

The `reference-parity-matrix --json` gate validates matrix shape and exposes `releaseClaimGateReady`; strict mode is intentionally reserved for public replacement-parity promotion and fails until the live measured benchmark row is full/passed.

The reference parity matrix remains the source of release-claim status.
Cursor/OpenCode degraded coverage blocks full host lifecycle claims until the
canonical capability matrix and fixture-backed smoke tests say otherwise.

- tool-routing fixture
- workspace-routing fixture
- runtime-routing fixture

## Agent-kit source repo JIT contract

Consumer repos use direct absolute `node <agent-kit>/bin/wp hook <name>` commands.
This source repo uses the same direct command shape, but setup/restore must be
run as:

```bash
WP_FORCE_SOURCE=1 wp setup repair
WP_FORCE_SOURCE=1 wp setup repair --restore-hooks
```

No managed shell wrappers are generated. `wp dev runtime-hooks enable` is the
only opt-in override for compiled/runtime hook dispatch; `wp dev runtime-hooks
disable` returns to JIT-first source dispatch.
