# Architecture: the three-layer model

Agent-kit splits the agent surface into three layers with different
ownership and lifecycle. Knowing which layer you're editing is the
difference between making a real change and silently losing your work
on the next sync.

## The layers

### Layer 1 — Catalog (read-only template)

**Location:** `node_modules/@webpresso/agent-kit/catalog/`

Ships with the package. Curated commands, skills, workflows, rules,
guides, and doc templates that agent-kit maintainers consider stable
defaults.

- **Owner:** agent-kit
- **Lifecycle:** updated by bumping the package version
- **Edit?** Never. Anything you write here disappears on `pnpm install`.

### Layer 2 — Repo-local agent surface (your fork)

**Location:** `.agent/` (in your repo)

Scaffolded by `ak setup` (`ak init` remains an alias), which copies selected catalog content into
`.agent/`. From that moment on the files belong to your repo: tracked
in git, editable, divergence-allowed.

- **Owner:** your repo (commit the changes)
- **Lifecycle:** committed alongside code; merged like any other change
- **Edit?** Yes — this is where local customizations live.

### Layer 3 — IDE-derived surfaces (regenerable)

**Location:** `.claude/rules/`, `.claude/skills/`, `.codex/agents/`,
`.cursor/rules/`, `.windsurf/skills/`, `.gemini/commands/`, `.agents/skills/`

Generated from `.agent/` by **`ak compile`** (v0.15+):

1. `ak compile` calls `flattenAgentDir('.agent/')` to build a `.rulesync/`-shaped
   directory, then delegates to [rulesync](https://github.com/dyoshikawa/rulesync)
   (`rulesync generate --targets claude,codex,cursor,gemini,opencode,windsurf`) to
   emit per-IDE surfaces. Agent-kit does not reimplement rulesync's emitters.
2. Four **plugin manifests** are also emitted and committed (not gitignored):
   `.claude-plugin/plugin.json`, `.codex-plugin/plugin.json`,
   `.cursor-plugin/plugin.json`, `gemini-extension.json`. These are ~30-line JSON
   files that allow IDE plugin marketplaces to install skills without file-writing.
3. The **AGENTS.md merger** runs after compilation, writing the repo-root `AGENTS.md`
   from the layered sources in `.agent/memory/`.

`ak compile` is idempotent — re-running with identical `.agent/` source is a no-op
(content-hash manifest at `.agent/.compile-manifest.json`).

Deliberately NOT generated:

- `.codex/prompts/`: deprecated by OpenAI (see the
  [Codex custom-prompts docs](https://developers.openai.com/codex/custom-prompts));
  was home-only (`~/.codex/prompts/`) anyway, never project-local.
- `.codex/skills/`, `.opencode/skills/`: redundant. Codex scans
  `.agents/skills/`, not `.codex/skills/`
  ([source](https://developers.openai.com/codex/skills)); OpenCode
  already walks both `.claude/skills/` and `.agents/skills/` as
  project-local fallbacks
  ([source](https://opencode.ai/docs/skills/)).

- **Owner:** auto-generated; do not edit by hand
- **Lifecycle:** regenerated by `ak compile` (idempotent). `ak sync` remains an alias for the legacy symlink path; prefer `ak compile` in new repos.
- **Edit?** No. Edit `.agent/` and re-run `ak compile`.

### Optional layer — Blueprint SQLite projection

**Location:** `.agent/.blueprints.db` (gitignored; rebuilt on cold start)

`ak blueprint db build` projects all `blueprints/**/_overview.md` and
`tech-debt/**/*.md` into a SQLite store. Agents query it via 8 MCP tools
(`ak_blueprint_query`, `ak_blueprint_task_next`, etc.) instead of reading 50KB
of markdown. Humans browse it with `ak blueprint db browse` (Datasette).

This layer is derived — markdown stays canonical. Never write back to SQLite
from a remote agent; commit the markdown change instead.


## Public package boundary

Agent Kit is maintained as a standalone public package. Keep the shipped
surface split into three bands:

- **Public core:** `ak setup`, `ak blueprint`, `ak sync`, `ak skill`,
  `ak test`, `ak e2e`, package exports, templates, and docs that work in
  any Node/Bun repo. These paths must not import `@webpresso/*` or assume
  `webpresso/` directories.
- **Optional integrations:** tool-specific generated surfaces such as
  `.claude/`, `.cursor/`, `.windsurf/`, `.opencode/`, `.agents/skills/`,
  and `.gemini/`. They are derived from `.agent/` and can be skipped by
  consumers that do not use a given tool.
- **Optional runtime adapters:** OMX/team/pll execution details belong behind
  a future `agent-kit-omx` seam, not in the public-core mental model.
- **Optional hook adapters:** reusable pretool validators belong behind a
  future `agent-kit-hooks` seam after Webpresso-specific policy is removed.
- **Internal-only host code:** `apps/cli-wp`, Webpresso product templates,
  host-owned E2E orchestration, and app-specific validator policy remain
  outside the package core. Public docs may mention them as consumers, but
  setup, blueprint lifecycle, symlink, test, and E2E commands must not
  require them.

Fresh consumer repos default to top-level `blueprints/` and `tech-debt/`.
The historical `webpresso/blueprints/` and `webpresso/tech-debt/` layouts
remain a compatibility fallback when the Webpresso sentinel (`webpresso/config.yaml`) exists.

To pin the location explicitly — bypassing directory detection — set `blueprintsDir` in
`.agent-kitrc.json`:

```json
{ "blueprintsDir": "webpresso/blueprints" }
```

This is the highest-priority override; all blueprint commands (`new`, `list`, `audit`,
`start`, `finalize`, `move`) and the pretool hook validator read from the same source.

Future `agent-kit-hooks` candidates are tracked in `docs/research/quality-audits/agent-kit-hooks-candidate-matrix.md`; only generic validators should move into that optional package.

## Why three layers and not two

Two alternatives fail for specific reasons:

1. **Install-and-use directly from `node_modules/`** — consumers can't
   customize without an "override" indirection, and upgrades silently
   replace local edits. The fork-and-edit model in `.agent/` sidesteps
   both.

2. **Symlink `node_modules/` straight into `.claude/`** — breaks the
   moment a second IDE (Gemini) needs a different file format. The
   TOML transform isn't a symlink; it's a build step. Once one IDE
   needs a transform, you need a canonical source (`.agent/`) to
   transform *from*.

## Lifecycle in practice

```
agent-kit ships an update
  └─→ pnpm update @webpresso/agent-kit        (catalog refreshed in node_modules/)
       └─→ npx ak setup --overwrite           (or: ak skills install <name>)
            └─→ new catalog content copied into .agent/   — review the diff
                 └─→ npx ak sync       (regenerate .claude/, .gemini/, …)
```

Skip `ak setup` after a package update and you keep running the old
`.agent/` content. Skip `ak sync` and your IDEs still see the
old surface. Both steps matter.

## Anti-patterns

- **Editing `node_modules/@webpresso/agent-kit/catalog/`** — gone on
  next install.
- **Editing `.claude/commands/`** — overwritten on next `ak sync`.
- **Forgetting to commit `.agent/`** — collaborators get the old
  surface or none at all.
- **`ak setup --overwrite` without diffing first** — wipes per-repo
  customizations.
